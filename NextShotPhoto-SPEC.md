# NextShotPhoto — 分镜尾帧 Prompt 生成器 设计规范

> **子 Skill 归属**：PerfectPhoto（`D:\skills\PerfectPhoto\`）
> **状态**：已实现 v1.0
> **日期**：2026-07-21

---

## 1. 概述

### 1.1 解决的问题

用户用 PerfectPhoto 完成了第一镜（首帧图）的 prompt 生成和出图后，需要对后续每个分镜（Shot）生成**尾帧图片的 prompt**，满足：

- 人物、产品、氛围**与上一镜保持一致**
- 每镜只变化镜头语言（景别/机位/运镜/动作）
- 首帧图 + 尾帧图可以通过 I2V 引擎（如 Seedance 2.0）的首尾帧模式插值生成视频

### 1.2 核心原则

1. **不从头问 8 步** — 只问变化量（Delta），不变域（Invariant）已锁定
2. **字面量焊接** — 锁定段每镜必须完全一致，不允许"同上"或缩写
3. **变体隔离** — 每镜只改 3-4 个字段，锁定段不可触碰
4. **冲突检查** — 用户的变化是否引入副作用，自动检测并追问

### 1.3 用户工作流全貌

```

### 1.4 多镜循环状态机

从 PerfectPhoto 完成后到多镜结束，Agent 运行一个**明确的状态机**。每轮状态决定 Agent 做什么、输出什么、等待什么输入。

```
                    ┌──────────────────────┐
                    │  S0: 等待用户意图      │
                    │  （PerfectPhoto 完成）  │
                    └──────┬───────────────┘
                           │ 用户说"做下一个镜头"/
                           │ "分镜"/"shot"
                           ▼
              ┌───────────────────────────┐
              │  S1: 图片链完整性检查       │
              │  🔁 每镜进入时都运行        │
              │                           │
              │  首次进入 → 检查 S0 首帧图  │
              │  后续每镜 → 检查上一镜尾帧   │
              │                           │
              │  ✅ 有图 → 不做任何提示     │
              │  ⚠️ 缺图 → 提示 + 继续走   │
              └──────┬────────────────────┘
                     │
                     ▼
              ┌───────────────────────────┐
              │  S2: 用户输入解析          │
              │  接收用户的分镜需求         │
              │  解析为 Delta              │
              │  景别: ?  机位: ?          │
              │  运镜: ?  动作: ?          │
              └──────┬────────────────────┘
                     │ 有缺失字段
                     ▼
              ┌───────────────────────────┐
              │  S3: DIM 追问              │
              │  追问缺失字段（最多2轮）     │
              │  → 如果用户信息足够，提前退出 │
              │  → 如果信息不足，补充追问    │
              └──────┬────────────────────┘
                     │ 信息收齐
                     ▼
              ┌───────────────────────────┐
              │  S4: 映射检查              │
              │  检查 Delta 副作用         │
              │  有冲突 → 追问/自动处理     │
              │  无冲突 → 下一步            │
              └──────┬────────────────────┘
                     │
                     ▼
              ┌───────────────────────────┐
              │  S5: 推导 + 输出           │
              │  变换矩阵 → 尾帧 prompt    │
              │  diff 校验 → 映射报告      │
              │  输出三者：                 │
              │  ① prompt                  │
              │  ② 校验报告                │
              │  ③ 更新后的 NextShotContext │
              └──────┬────────────────────┘
                     │
                     ▼
              ┌───────────────────────────┐
              │  S6: 可选图片生成           │
              │  「要用XX生成尾帧图吗？」    │
              │  是 → 检测参考图:           │
              │    有 history[N-1].image →  │
              │      调用 tool(prompt+参考图)│
              │    无 → 纯文生图             │
              │      调用 tool(prompt)       │
              │  否 → 跳过                  │
              │  → 记录图片到 history[N]    │
              └──────┬────────────────────┘
                     │
                     ▼
              ┌───────────────────────────┐
              │  S7: 等待下一镜            │
              │  输出引导语：              │
              │  「下一镜想怎么拍？」        │
              │  用户说"下一个镜头：..."    │
              │  → 回到 S2                 │
              │  用户说"回到第N镜"         │
              │  → 走 §7.4 分支回退        │
              │  用户说"不拍了"            │
              │  → 退出循环，展示分镜汇总   │
              └────────────────────────────┘
```

**关键设计原则**：

1. **S2→S3→S4 是单次 DIM 交互**（§3），每镜最多 3 轮问答。用户说话如果够完整（"推到产品特写，俯拍45度，人物拿起杯子"），可以直接从 S2 跳到 S5，跳过 S3。

2. **S6 是可选后处理**，不影响推导，不改变状态。生成完直接回到 S7。

3. **S7→S2 是自动循环**，不需要用户重新输入命令。用户只说「下一个镜头：推到特写」就自动解析，不需要再说「NextShotPhoto」。

4. **S1 每镜都运行**，但行为不同：

   - **首次进入（Shot 1 尾帧推导前）**：检查 `history[0].image`（首帧图）
     - 空 → 「还没有首帧图。要用XX生成一张吗？已有 prompt。」是→生成并记录；否→用户提供路径；跳过→参考链断开
     - 非空 → ✅ 继续
   
   - **后续每镜（Shot N 尾帧推导前）**：检查 `history[N-1].image`（上一镜尾帧图）
     - 空 → ⚠️ **不阻塞，仅提示**：「上一镜尾帧图未生成，本镜 prompt 推导不受影响。但后续生成尾帧图时只能用纯文生图（无参考图），一致性效果略弱于图生图模式。」
     - 非空 → ✅ 继续（S6 生成时可传入参考图）
   
   **关键：S1 只提示，不阻塞。** 用户可以在任何镜跳过生成图片，推导流程不受影响。

5. **任何时候用户说"不拍了"** → 退出循环，输出分镜汇总表 + 批量导出选项：

```
📋 分镜汇总（共 5 镜）
┌───────┬────────┬─────────┬────────┬────────────────┬──────────┐
│  Shot │ 景别   │ 机位    │ 运镜   │ 动作          │ 有图？  │
├───────┼────────┼─────────┼────────┼────────────────┼──────────┤
│   1   │ 中景   │ 平视    │ 固定   │ 手轻握咖啡杯   │ ✅       │
│   2   │ 近景   │ 俯拍45° │ 慢推   │ 拿起咖啡杯闻香 │ ✅       │
│   3   │ 极特写 │ 平视    │ 固定   │ 咖啡杯蓝色花纹 │ ❌       │
│   4   │ 中景   │ 过肩   │ 环绕   │ 人物翻看旧书  │ ✅       │
└───────┴────────┴─────────┴────────┴────────────────┴──────────┘

🔄 导出选项：
  [1] 导出所有 prompt（文本文件）
  [2] 导出所有已生成图片路径（文件列表）
  [3] 两者都要
```

用户选择后，Agent 生成对应导出文件。如果用户未指定导出，默认输出纯文本汇总。shot.index 从 Shot 0（首帧）开始计数，Shot 1 起为尾帧。

```
📋 分镜汇总（共 5 镜）
┌───────┬────────┬─────────┬────────┬────────────────┐
│  Shot │ 景别   │ 机位    │ 运镜   │ 动作          │

## 2. 镜头上下文数据结构

这是整个系统的核心——跨镜头传递的结构化数据，而非像素级状态。

### 2.1 结构定义

```yaml
NextShotContext:
  # ======== 锁定域（跨镜头不变量） ========
  lock:
    character: "瓜子脸，清冷气质，明确成年，黑色齐肩直发，清透裸妆"
    clothing: "白色真丝衬衫，深灰高腰阔腿裤，简约银链"
    products:                         # 多产品支持，列表
      - id: "coffee_cup"
        name: "白瓷咖啡杯，蓝色花纹"
        position: "放在木托盘上"       # 默认位置
        interaction: null             # 当前互动方式（由 action 覆盖）
      - id: "book"
        name: "精装旧书，棕色皮面"
        position: "桌上摊开"
        interaction: null
    atmosphere: "暖金逆光，胶片颗粒，低对比，亮部轻微溢出"
    imaging: "尼康相机摄影，自然光质感，细腻色彩"
    lighting: "窗边自然光从侧后方透入，白雾逆光在背景形成柔和光晕"

  # ======== 当前镜头域（每镜变化） ========
  shot:
    index: 2                       # 当前镜次
    scene: "远景全身"               # 景别
    angle: "平视"                   # 机位
    movement: "固定"                # 运镜
    action: "人物坐在桌前，手轻握咖啡杯"  # 动作
    foreground: null                # 前景（可选）
    active_product: "coffee_cup"    # 当前镜头主交互的产品 ID

  # ======== 产品互动追踪（辅助映射检查） ========
  # 由 action 自动更新，不需要用户手动维护
  # 规则：action 提到哪个 product id，就更新该 product 的 interaction
  # 示例：action="人物翻开了棕色皮书" → book.interaction = "人物手中翻看"

  # ======== 历史记录（参考用） ========
  history:
    - shot: 0                        # Shot 0 = 首帧（PerfectPhoto 产出）
      prompt: "[首帧完整 prompt 文本]"
      image: "[首帧图片路径/URL]"     # 实际生成的图片路径
      source: "PerfectPhoto"          # 记录来源（PerfectPhoto / ImageGen / 用户提供）
      branch_id: "main"
    - shot: 1
      prompt: "[完整 prompt 文本]"
      image: "[尾帧图片路径/URL]"     # 本镜生成的尾帧图片
      source: "ImageGen"
      tail_frame: "[文件路径/URL]"
      branch_id: "main"
```

### 2.2 锁定域不可改字段

| 字段 | 保护级别 | 说明 |
|------|---------|------|
| `lock.character` | 🚫 不可改 | 脸型、发型、妆容、气质 |
| `lock.clothing` | 🚫 不可改 | 服装款式、颜色、材质、配饰 |
| `lock.products[]` | ⚠️ 位置可微调 | 多产品列表。每个产品的**外观/颜色/纹理**不可变，**位置和互动方式**可随动作变化 |
| `lock.atmosphere` | 🚫 不可改 | 色调、颗粒感、对比度风格 |
| `lock.imaging` | 🚫 不可改 | 成像设备、模拟胶片 |
| `lock.lighting` | 🚫 不可改 | 光源方向、光线质量 |

> ⚡ **位置微调例外**：产品因为动作从桌面移到人物手上——这是合理的，不需要改 `lock.products[].name`，只需要更新 `lock.products[].interaction` 并在 `shot.action` 中描述。  
> ⚡ **多产品互动规则**：action 描述中 `@产品ID` 标记当前互动的产品。例如「人物翻开了 @book」→ 自动将 `book.interaction` 更新为「人物手中翻看」，其他产品的 `interaction` 保持不变。  
> ⚡ **默认产品**：`shot.active_product` 指向当前镜头的主要互动产品。如果未指定，默认选择第一个产品。

## 3. DIM 交互框架

### 3.1 Delta（变化量）

分镜需求的 4 个 Delta 槽位：

| 槽位 | 英文 | 示例值 | 判断标准 |
|------|------|--------|---------|
| 景别 | Shot Scale | 远景/全景/中景/近景/特写/极特 | 上一镜是什么 |
| 机位 | Camera Angle | 平视/低机位/俯拍/仰拍/过肩/侧面/背后 | 是否改变 |
| 运镜 | Camera Movement | 固定/推/拉/摇/移/跟 | 是否改变 |
| 动作 | Action | 人物在做什么 + 产品如何互动 | 是否改变 |

### 3.2 Invariant（不变域）

锁定域 6 个字段不需要问，但用户主动提出要改时需要：

- 用户说「人物换成红色的衣服」→ ⚠️ 违反 `lock.clothing` 不可改，拦截确认
- 用户说「背景从书房切到阳台」→ 这实际是一个**新场景**，需要评估是否影响氛围锁定

### 3.3 Mapped（映射检查）

用户填了 Delta 后，自动检查副作用：

| 变化 | 潜在副作用 | 触发条件 | 追问示例 |
|------|-----------|---------|---------|
| 景别→极特写 | 上一镜前景元素可能不适用 | 景别变窄 |「现在是极特写，上一镜的花丛前景要保留还是去掉？」|
| 动作→跑动 | 服装流态需要补充 | 动作幅度大 |「跑动时裙摆会被风吹起，需要在 prompt 里补充动态描述吗？」|
| 机位→俯拍 | 原锁定光线方案可能失效 | 机位角度变化>30° |「俯拍时原来的逆光方案还在吗？还是需要调整？建议保持锁定的逆光不变，只调整人物面部补光。」|
| 产品位置变化 | 从桌上→手上，需要确认互动逻辑 | `product_position` 改变 |「产品从木托盘上移到人物手上，是用右手拿杯身还是左手托杯底？」|

### 3.4 用户输入 → 解析 → 追问 映射表

```
用户说「推到产品特写」
→ 解析：
   景别=特写 ✓   运镜=推 ✓   机位=✗   动作=✗
→ 追问：
   ① 平视角看产品，还是俯拍桌面，还是过肩看？
   ② 产品在人物手上还是放在桌上？
```

```
用户说「人物拿起咖啡杯，低头闻香，近景」
→ 解析：
   动作=完整 ✓   景别=近景 ✓   机位=✗   运镜=✗
→ 追问：
   ① 这个角度拍人物正面还是侧面？
   ② 镜头固定还是跟着动作微移？
```

```
用户说「换个角度拍」
→ 评语：信息不足
→ 反问：「好，上一镜是平视正面中景。这一镜你想怎么变——景别推近？机位换侧面？还是运镜动起来？」
```

## 4. 三重锁定防漂移

### 4.1 第一重：字面量焊接

每镜 prompt 中的锁定段必须是**逐字逐符一致**的复制。

**实现**：在输出 prompt 前，将当前镜的锁定段与上一镜的锁定段做字符串 diff：

```
✅ PASS: 5 行锁定段与 Shot 1 完全一致
❌ FAIL: 第 3 行「白色真丝衬衫」→「白色丝绸衬衫」
→ 拦截，修正后再输出
```

### 4.2 第二重：变体隔离

输出 prompt 分两个区域：

```
============ 锁定域（不变） ============
[lock.character]
[lock.clothing]
[lock.products 各条目]
[lock.atmosphere]
[lock.imaging]
[lock.lighting]
============ 变体域（本镜变化） ============
景别：{shot.scene}
机位：{shot.angle}
运镜：{shot.movement}
动作：{shot.action}
============ 负面约束 ============
不要因为景别变化改变产品颜色和花纹
不要出现不存在的第二只手
[继承并可能调整]
```

**检查规则**：变体域字数 ≤ 总 prompt 的 15%。超出说明变了太多，警告确认。

### 4.3 第三重：首帧视觉锚定（视频引擎层保障）

当用户用 Seedance 首尾帧模式生成视频时：

- 上一镜的尾帧图 = 本镜视频的**首帧**
- 新生成的尾帧图 = 本镜视频的**尾帧**
- Seedance 会在首帧和尾帧之间插值

即使用户的尾帧 prompt 生成的图片有轻微漂移，Seedance 也因为首帧图已经固定了人物/产品/颜色而把视频拉回来。**文字约束 + 视觉锚定，双重保障。**

## 4.5 尾帧推导核心算法

这是 NextShotPhoto 的核心智能——**给定首帧的结构化字段 + 运镜选择 + 动作描述，如何推导出尾帧的字段**。本质上是一个**结构化变换函数**，不是文本改写。

### 4.5.1 基本原理

尾帧不是凭空生成的，它是首帧在经过「一段运镜 + 一段动作」之后自然抵达的画面状态。因此尾帧的每个结构化字段可以通过一个变换函数从首帧推导得出：

```
tail_prompt = TRANSFORM(first_context, camera_movement, shot_action)
                                  ↓
                        对每个字段应用变换规则
```

其中：

- `first_context` = 上一镜的结构化上下文（含锁定域 + 当前镜头域）
- `camera_movement` = 用户选择的运镜类型（推/拉/环绕/摇/跟随……）
- `shot_action` = 用户在镜头内的动作描述

### 4.5.2 运镜变换矩阵

每个运镜类型定义了一组**字段级变换规则**——哪些字段变，变成什么：

```
运镜类型：推镜头 (Push In)
  变换规则：
    shot.scene:      中景 → 近景（景别向主体推进一级，压迫感增强）
    foreground:      因镜头推进而放大虚化，占据更多画面
    background:      被压缩裁切，部分元素出画
    depth:           景深变浅，背景虚化增强，焦点更集中
    composition:     主体周围空间减少，环境信息减弱
  运镜描述短语：    "镜头平滑向前推进，从[首帧景别]推至更近"
  映射检查：
    - ⚠️ 产品特写时确认产品不被裁切出画
    - ⚠️ 推进幅度过大可能丢失前景元素

运镜类型：拉镜头 (Pull Out)
  变换规则：
    shot.scene:      近景 → 中景（景别向环境方向退一级）
    foreground:      因拉远而缩小，更多环境元素入画
    background:      更多空间显现，环境感增强，视野扩展
    depth:           景深变大，背景清晰度提高
    composition:     主体周围空间增大，环境信息增多
  运镜描述短语：    "镜头平滑向后拉远，从[首帧景别]拉出展现更多环境"
  映射检查：
    - ⚠️ 新入画元素不应与锁定域冲突（出现不该出现的东西）

运镜类型：环绕 (Orbit)
  变换规则：
    shot.angle:      平视正对 → 移动到主体侧面/斜后方
    foreground:      环绕过程中前景物体从一侧到另一侧
    background:      根据环绕角度完全改变（从 A 场景转到 B 场景）
    lighting:        !! 需要检查——原锁定光源方向可能随视点变化失效
    composition:     主体从正面展示变为侧面/背面展示
  运镜描述短语：    "镜头围绕主体做[半圆/全圆]平滑环绕运动"
  映射检查：
    - ⚠️ 光线映射：环绕时逆光方案可能失效，需补侧面光
    - ⚠️ 人物正面特征不可见时，确认是否接受侧脸/背面

运镜类型：手持晃动 (Handheld)
  变换规则：
    stability:       固定/平稳 → 轻微呼吸晃动，自然防抖
    composition:     画面有随机轻微偏移，非固定构图
  运镜描述短语：    "手持拍摄风格，镜头带有自然轻微呼吸抖动"
  映射检查：
    - 手持与极特写组合时抖动感需减弱（太近时晃得看不清）

运镜类型：摇镜头 (Pan)
  变换规则：
    shot.angle:      注意力从左到右/从右到左扫视
    foreground:      镜头扫过时前景物体形成动态模糊
    background:      从场景 A 部分过渡到场景 B 部分
  运镜描述短语：    "镜头水平向右/向左缓慢摇动，逐步揭示新空间"
  映射检查：
    - ⚠️ 摇镜头以「揭示」为目的，需确认新区域无穿帮

运镜类型：跟随 (Follow)
  变换规则：
    action:          人物向[方向]移动，镜头保持同步速度跟随
    foreground:      移动过程中前景连续变化
    background:      随人物移动连续变化的空间序列
  运镜描述短语：    "镜头与主体保持同步速度，跟随人物向[方向]移动"
  映射检查：
    - ⚠️ 移动路径需连续，不能跳跃空间

运镜类型：希区柯克变焦 (Dolly Zoom)
  变换规则：
    subject_size:    主体在画面中大小不变（核心特征）
    background:      因推拉+变焦的共同作用，背景被强烈压缩/拉伸
    emotion:         空间扭曲感，心理压迫，震惊效果
  运镜描述短语：    "进行希区柯克变焦，摄像机后拉同时长焦推进，主体保持大小"
  映射检查：
    - 此为强风格化运镜，运镜本身即为画面主题，需用户确认是否匹配镜头功能

运镜类型：无人机穿越 (FPV Drone)
  变换规则：
    shot.angle:      固定机位 → 自由飞行视角，第一人称
    foreground:      高速移动中前景急速掠过，产生强烈动态模糊
    background:      连续穿越空间，背景快速变化，空间感强烈
    depth:           景深较大，远近皆清晰，增强临场感
    stability:       剧烈晃动，模拟无人机飞行姿态
  运镜描述短语：    "FPV 无人机视角，高速穿梭，大幅度滚转与甩尾运镜，动态模糊"
  映射检查：
    - ⚠️ FPV 视角下人物面部不可见，人物锁定字符仍需维持（可出自拍画面之外的语音/叙事）
    - ⚠️ 动态模糊过强时产品细节丢失，需在负面约束中补充「产品轮廓保持清晰」

运镜类型：子弹时间 (Bullet Time)
  变换规则：
    scene:           动态 → 瞬间凝固定格
    action:          运动过程 → 时间近乎凝固，周围环境静止
    background:      背景中的人物/物体/粒子保持静止
    camera:          镜头在凝固的时空中缓慢移动/环绕
  运镜描述短语：    "画面突然进入子弹时间（Bullet Time），时间近乎凝固，镜头在静止空间中缓慢移动"
  映射检查：
    - ⚠️ 子弹时间是强风格化效果，需确认是否匹配镜头叙事功能
    - ⚠️ 人物表情在凝固瞬间被放大，需确保表情自然

运镜类型：快速变焦 (Snap Zoom)
  变换规则：
    shot.scene:      中景/近景 → 瞬间推进至极特写
    action:          常规动作 → 在变焦终点处突然定格
    focus:           画面瞬间切换焦点到[指定部位/道具]
    emotion:         冲击感，强调，惊吓
  运镜描述短语：    "极速光学变焦推镜头，在[焦点对象]处突然定格"
  映射检查：
    - ⚠️ 瞬间变焦模式下产品不能出现在画面边缘（会被裁切）
    - 极特写时需加强手指负面约束

运镜类型：闪摇 (Whip Pan)
  变换规则：
    shot.angle:      场景 A → 极速水平扫过 → 场景 B
    foreground:      扫过时产生强烈横向动态模糊
    background:      从一个空间切换到另一个空间
    motion_blur:     全程模糊，仅起点和终点清晰
  运镜描述短语：    "镜头快速水平甩动，产生合理的运动模糊，从[场景A]切换到[场景B]"
  映射检查：
    - ⚠️ 闪摇的终点场景需用户指定，不能默认「回到原位」
    - ⚠️ 闪摇不适合室内狭小空间（横向距离不够）

运镜类型：荷兰角 (Dutch Angle)
  变换规则：
    shot.angle:      水平 → 向一侧倾斜 15-30 度
    composition:     地平线打破，画面失衡，不安感
    emotion:         心理失衡，诡异，紧张
  运镜描述短语：    "画面呈现荷兰式倾斜（Dutch Angle），镜头向一侧倾斜[角度]度"
  映射检查：
    - ⚠️ 荷兰角不适用于产品展示（会让产品看起来歪斜）
    - ⚠️ 建议超过 30 度会过于刻意，20 度为通用安全值

运镜类型：组合序列 (Multi-Shot Sequence)
  变换规则：
    sequence:        [运镜1] + [运镜2] + [运镜3] 串联
    scene:           随每个子运镜分段变化
    action:          随序列推进逐步进展
    composition:     首帧 → 过渡帧序列 → 尾帧
  运镜描述短语：    "多镜组合序列：[运镜1描述] → [运镜2描述] → [运镜3描述]"
  映射检查：
    - ⚠️ 组合序列需用户指定子运镜顺序和时长占比
    - ⚠️ 序列长度不宜超过 3 个子运镜，否则模型难以稳定跟随

### 4.5.3 动作驱动的字段覆盖

当用户指定了动作（action），动作描述会**覆盖**运镜变换中的动作默认值，并根据 `@产品ID` 标记自动更新对应产品的互动状态。运镜变换先决定"机位怎么动"，动作描述再决定"人物/产品怎么变"：

```
输入：
  运镜 = 推镜头 (Push In)
  动作 = "人物拿起 @coffee_cup，低头闻香"

推导步骤：
  ① 运镜变换：景别 中景 → 近景，背景压缩，景深变浅
  ② 动作覆盖：shot.action = "人物拿起咖啡杯，低头闻香"
     → 自动识别 @coffee_cup，更新 products[0].interaction = "人物右手握着"
     → 其他产品（如 @book）的 interaction 保持不变
  ③ 映射检查：产品从「放在木托盘上」变为「人物手中」，已覆盖

### 4.5.4 推导流程（完整）

```
输入：
  ① first_context（来自上一镜）
  ② camera_movement（用户选或系统推荐）
  ③ shot_action（用户描述，可选）

Step 1: 复制锁定域（不变）
  tail.lock = first_context.lock  — 逐字符复制，不做任何修改

Step 2: 复制当前镜头域
  tail.shot = first_context.shot

Step 3: 应用运镜变换矩阵
  FOR each rule in CAMERA_MOVEMENT_TABLE[camera_movement].transforms:
      IF rule.field matches shot field:
          tail.shot[field] = apply(first_context.shot[field], rule)

Step 4: 如果用户指定了动作，覆盖 action 字段
  IF shot_action is not empty:
      tail.shot.action = shot_action
      CALL update_product_position(shot_action) — 检查产品位置是否变化

Step 5: 运镜描述组装
  motion_phrase = CAMERA_MOVEMENT_TABLE[camera_movement].motion_desc
  → 植入 prompt 变体域

Step 6: 映射检查
  FOR each check in CAMERA_MOVEMENT_TABLE[camera_movement].mapping_checks:
      IF check triggers on current state:
          ADD to issues list (追问用户或自动处理)

Step 7: 组装完整 prompt + 输出映射报告
  prompt = assemble(tail, motion_phrase)
  output = prompt + mapping_report

输出：
  ① 尾帧 prompt（含锁定域+变体域+运镜描述+负面约束）
  ② 映射检查报告（无/警告/需追问）
  ③ 更新后的 NextShotContext
```

### 4.5.5 完整推导示例

```
首帧（Shot 1 尾帧上下文）：

### 4.5.6 关键设计原理

为什么运镜变换矩阵是对的：

1. **运镜是确定性变换** — 推镜头就一定会让景别变近、背景压缩、景深变浅。这不是风格选择，是物理规则。所以变换规则可以硬编码。

2. **动作是自由度** — 运镜锁死机位怎么动，动作给用户自由度决定里面的人/产品怎么动。两不相欠。

3. **映射检查捕获副作用** — 环绕会改变光源方向，手持会让极特写看不清——这些不是规则出了问题，是规则组合产生的新约束。映射检查就是干这个的。

4. **锁定域一步不动** — 变换矩阵只作用于 `shot.*` 字段。`lock.*` 字段在任何运镜下都不变，这是防漂移的第一道防线。

### 4.5.7 运镜感知的负面约束引擎

不同运镜组合下，负面约束需要自动调整——手持运镜下「不要晃动」是矛盾的，极特写下「手指形态自然」比中景更关键。每镜的负面约束不是简单继承上一镜的 list，而是根据当前运镜+景别+动作计算。

**约束增益/衰减规则**：

| 运镜/景别 | 影响的负面约束 | 操作 | 原因 |
|-----------|---------------|------|------|
| 手持晃动 | 「构图平稳」「不要抖动」 | 🗑️ 删除 | 手持=抖动是特征不是缺陷 |
| 极特写 | 「手指形态自然」「不要多余手指」 | 🔺 增强 | 特写放大手指，失败概率倍增 |
| FPV 无人机 | 「产品轮廓保持清晰」 | ➕ 新增 | 动态模糊会吞掉产品细节 |
| 快速变焦/Snap Zoom | 「定格画面不要模糊」 | ➕ 新增 | 骤停时刻需要绝对清晰 |
| 荷兰角 | 「地平线保持水平」 | 🗑️ 删除 | 倾斜就是荷兰角的目的 |
| 子弹时间 | 「不要定格动作」 | 🗑️ 删除 | 定格是子弹时间的核心效果 |
| 环绕运镜 | 「人物背面/侧面角度正常」 | ➕ 新增 | 环绕必然出现非正面视角 |
| 慢动作 | 「不要运动模糊」 | 🗑️ 删除 | 慢动作可能伴随轨迹模糊 |
| 人物手持产品 | 「产品不要出现在非指定位置」 | ➕ 新增 | 手部互动时产品位置易偏移 |
| 多人交互 | 「前景背影与主体分离清晰」 | ➕ 新增 | 多人时前景遮挡易混淆 |

**实现方式**：在推导流程 Step 6（映射检查）之后、Step 7（组装）之前插入一个约束计算步骤：

```
Step 6.5: 计算运镜感知的负面约束
  negative_constraints = inherit_from_previous()  # 继承上一镜基础约束
  
  FOR each rule in NEGATIVE_CONSTRAINT_TABLE:
      IF rule matches current camera_movement + shot.scene + shot.action:
          APPLY rule.operation to negative_constraints:
              case "🗑️ 删除": remove constraint from list
              case "🔺 增强": increase priority of constraint
              case "➕ 新增": add constraint to list
  
  # 去重：相同含义的约束只保留最强的一条
  negative_constraints = deduplicate(negative_constraints)
```

**示例**：

```
上一镜负面约束：
  不要塑料皮肤，不要手指畸形，不要多余肢体
  不要大幅扭头，不要真实品牌logo，构图平稳

当前运镜：手持晃动 + 人物手持咖啡杯
→ 自动调整：
  删除「构图平稳」（手持时抖动正常）
  新增「产品在手中位置稳定，不因晃动脱手」
  新增「咖啡杯内液体不溢出」
  其他约束继承不变

输出负面约束：
  不要塑料皮肤，不要手指畸形，不要多余肢体
  不要大幅扭头，不要真实品牌logo
  产品在手中位置稳定，不因晃动脱手       # 🆕 手持特有
  咖啡杯内液体不溢出                      # 🆕 手持特有
```

同样更新推导流程 Step 6.5：

```
Step 6:  映射检查
Step 6.5: 负面约束计算 ← 新插入
  输入：当前运镜 + 景别 + 动作
  输出：运镜感知的负面约束列表
Step 7:  组装完整 prompt（含负面约束块）
```

### 5.4 可选图片生成集成

#### 5.4.1 设计动机

NextShotPhoto 是纯 prompt 层（§12.2），核心推导不涉及任何图片/视频生成。但输出 prompt 后，用户大概率需要用这个 prompt 去生成图片。如果在同一对话中**直接调用生成工具**完成这步，用户不需要离开当前上下文，而且可以自动传入上一镜的图片作为参考图，进一步降低漂移。

#### 5.4.2 触发时机

在「输出尾帧 prompt」（§5.1/5.2）完成后，追加一个可选步骤：

```
完成尾帧 prompt 推导 → 输出 prompt
  ↓
检测是否有可用的图片生成 skill：
  ① 用户曾说过用哪个工具（记忆在 NextShotContext 中）
  ② 如果不知道，问一次「你用的是哪个图片生成工具？」
  ③ 记录下来供后续自动调用
  ↓
询问：「要用 [工具名] 生成尾帧图吗？已有上一镜的图作为参考。」
  ↓
是：
  ① 参考图 = history[上一镜].image
  ② 调用生成 tool({ prompt: 尾帧prompt, image: 参考图 })
  ③ 记录生成的图片路径到 history[本镜].image
  ④ 告知用户图片已生成，路径已记录
否：跳过，只输出 prompt
```

#### 5.4.3 首帧图片的兜底

进入 NextShotPhoto 之前，`history[0].image` 可能为空（用户没有用当前对话生成首帧图）。此时：

```
检测 history[0].image 是否为空：
  空 → 询问「还没有首帧图。要用 [工具名] 生成一张吗？prompt 已经有了。」
       是 → 调用生成 tool({ prompt: history[0].prompt })
            → 记录到 history[0].image
       否 → 问「那你已经有图片吗？提供路径，我记录为参考图。」
            用户提供 → 记录到 history[0].image
            用户跳过 → 不记录，后续无法用参考图模式
  非空 → 正常走后续流程
```

#### 5.4.4 参考图传入规则

调用图片生成 tool 时，是否传入 `image` 参数取决于：

| 条件 | 行为 | 原因 |
|------|------|------|
| `history[上一镜].image` 存在 | ✅ 传入作为参考图 | 视觉锚定，降低漂移 |
| `history[上一镜].image` 不存在 | ❌ 不传，纯文生图 | 无可用参考 |
| 本镜是 Shot 1（首镜尾帧） | ✅ 传入 `history[0].image`（首帧图） | 首帧=上一镜 |
| 用户明确要求「不用参考图」 | ❌ 不传 | 尊重用户选择 |

#### 5.4.5 支持的生成工具

| 工具 | 支持参考图 | 调用方式 |
|------|-----------|---------|
| ImageGen | ✅ | `DeferExecuteTool({toolName:"ImageGen", params:{prompt, image}}) |
| gpt-image-2 | ✅ | 多模态输入，传 prompt + image_url |
| DragonCode | ✅（底层=gpt-image-2） | 同 gpt-image-2 |

用户首次使用时会问：「你用的是哪个图片生成工具？」记录到 `NextShotContext.generation_tool`，后续不再问。

#### 5.4.6 集成不影响核心推导

```
推导核心（§4.5）→ 输出 prompt → [可选] 调用生成工具
    ↑ 纯逻辑                ↑ 纯文本         ↑ 编排层
```

修改 §12.2 的职责边界，增加一条：

```

│                                     │
│  ✅ 接收/维护 NextShotContext        │

## 6. 自动检查清单

输出 prompt 前自动执行，全部通过才交付：

| 检查项 | 规则 | 失败处理 |
|--------|------|---------|
| 锁定段 diff | 与上一镜完全一致（字符级） | 拦截，自动修正 |
| 变体字段白名单 | 只改 [景别][机位][运镜][动作][前景] | 拦截，标出非法更改 |
| 变体字数比例 | 变体域 ≤ 15% 总字数 | 警告，建议保留更多锁定域 |
| 图片链完整性 | 上一镜图片存在或已标记跳过 | ⚠️ 提示但不阻塞（见 §1.4 S1） |
| 负面约束完整性 | 包含「产品颜色不变」「不多余肢体」 | 自动补全 |
| 负面约束运镜一致 | 负面约束不与当前运镜矛盾（如手持时无「构图平稳」） | 自动删除矛盾项（见 §4.5.7） |
| 产品互动一致性 | 动作中 @产品ID 对应的产品存在且互动描述合理 | 拦截，标出未定义的产品 ID |
| 映射触发 | 机位/动作变化是否触发副作用 | 如有则追问用户 |

## 7. 边界情况处理

### 7.1 用户说「这一镜不要锁定了，我想跳到另一个场景」

- 定位：这是一个**断链（break chain）** 操作
- 行为：重置锁定域，但保留历史记录
- 话术：「好，从这镜开始是一个新拍摄方案。锁定域会重置，但上一镜的上下文会保留作为参考。请描述新场景的 8 个维度。」

### 7.2 用户直接丢一张参考图过来说「按这个风格做分镜」

- 不走 PerfectPhoto 8 步，而是走**VLM 分支**（已设计但非主流程）
- 条件：如果用户启动 NextShotPhoto 时没有 PerfectPhoto 的上下文，自动切换到 VLM 模式
- 不推荐，但保留这个 fallback

### 7.3 用户说「上一镜我其实拍了张图，你帮我对齐」

- VLM 读图 + 反推结构化描述
- 然后与当前上下文做 diff，有冲突时让用户选择
- 这也算一条分支路径，但不是 MVP

### 7.4 多分支回退（Branch & Rollback）

实际一定会出现：用户做到第 5 镜不满意，想回退到第 3 镜重新推导。

**支持的操作**：

| 操作 | 触发词 | 行为 |
|------|--------|------|
| 回退到指定分镜 | `回到第 N 镜` | 从 `history[]` 恢复第 N 镜的 NextShotContext，丢弃 N+1 之后的历史 |
| 从指定分镜分支 | `从第 N 镜分一个新支` | 复制第 N 镜的 NextShotContext 作为新分支起点，分支需要命名（如「分支A」） |
| 查看分镜树 | `看看我的分镜` | 展示当前分镜树拓扑（含分支标记） |
| 切换分支 | `切到分支B` | 加载另一个分支的最新 NextShotContext |

**数据结构扩展**：

```yaml
NextShotContext:
  # ... 原有字段 ...
  
  # ======== 分支管理 ========
  branch_id: "main"                 # 当前分支标识
  branch_history:                    # 分支记录
    - shot: 3
      branch_id: "main"
      forked_at: null                # 从哪一镜分支出来的
    - shot: 5
      branch_id: "alt-angle"         # 在第 3 镜分出的分支
      forked_at:
        source_branch: "main"
        source_shot: 3
        context_snapshot: "[第 3 镜的完整 NextShotContext]"
```

**实现规则**：
1. 回退 = 从 `history[]` 中截取到目标镜为止，`shot.index` 接续
2. 分支 = 复制当前 Context 到新 `branch_id`，保留原始 `history[]` 不变，新分支的 `history[]` 重新开始记录
3. 切换分支 = 加载目标分支的最新 Context，但不丢失其他分支的记录
4. 回退后重新推导 = 从回退点的 Context 开始重新走 DIM 交互（§3），与正常流程完全一致

**交互示例**：
```
用户：回到第 3 镜，从那里重新来
系统：✅ 已回退到 Shot 3，当前状态如下：
      [显示 Shot 3 的 NextShotContext]
      现在是 Shot 4，后面新生成的会覆盖 Shot 4-5 的历史。
      请告诉我新镜头的需求。

用户：在第 3 镜分一个新支，叫「特写路线」
系统：✅ 已从 Shot 3 (main) 分支出新分支「特写路线」
      当前在「特写路线」分支，Shot 4，历史从 Shot 3 重新开始。
      原分支「main」有 Shot 4-5 的记录，随时可以切回去。
```

**交互引导**：
```
📂 分镜树
main:    Shot 1 → Shot 2 → Shot 3 → Shot 4 → Shot 5
                                    └── 特写路线: Shot 4 (你现在在这里)
```

## 11. 大师运镜库（Master Camera Movement）

### 11.1 设计动机

PerfectPhoto 有 13 位摄影大师预设（上田义彦/王家卫/森山大道等），控制视觉风格 + 光线 + 色彩。NextShotPhoto 做对应的**大师运镜库**，控制镜头运动 + 机位体系 + 转场逻辑。

| PerfectPhoto | NextShotPhoto |
|-------------|---------------|
| 摄影大师 | 电影导演/摄影师 |
| 成像方式 → 自动对齐 | 运镜方式 → 自动对齐 |
| 光线设计 → 按大师裁切 | 机位体系 → 按导演裁切 |
| 色彩后期 → 预设色调曲线 | 转场逻辑 → 按导演裁切 |

### 11.2 大师运镜预设

**交互形式**：用户进入 NextShotPhoto 后的第一步可选「选一位导演跟拍」，用法与 PerfectPhoto 的大师选择完全一致。

### 11.4 运镜词库（参考来源）

以下词汇可植入 prompt 变体域的「运镜」字段：

**景别词**：远景 / 全景 / 中景 / 近景 / 特写 / 极特 / 过肩 / 双人 / 骑轴

**运动词**：推 / 拉 / 摇 / 移 / 跟 / 环绕 / 升降 / 甩 / 晃 / 穿越 / 俯冲 / 爬升 / 横滚 / 悬停 / 漂移 / 急停

**节奏词**：缓慢 / 平滑 / 快速 / 极速 / 匀速 / 变速 / 骤停 / 慢动作 / 抽帧

**辅助词**：动态模糊 / 呼吸感 / 视差 / 焦平面 / 浅景深 / 边缘虚化 / 透视压缩 / 空间扭曲

### 11.5 与 DIM 交互的整合

选大师 → 影响追问选项裁切 → 影响输出 prompt 的运镜描述 → 影响映射检查预判

```
用户进入 NextShotPhoto
  ↓
[可选] 选大师运镜
  ↓
如果没有选，继续原 DIM 追问（§3）
如果选了，追问时优先按大师风格推荐运镜选项
  ↓
输出时自动植入大师风格短语
  ↓
映射检查增加大师风格相关的副作用检查
```

**示例**：用户选了斯皮尔伯格，说「下一个镜头：两人对话」
→ 追问：「斯皮尔伯格风格的话，推荐用环绕移焦（围绕两人做半圆运动，焦点在说话人之间切换）还是标准过肩反打？或者你有自己的想法？」
→ 输出 prompt 变体域：「机位：半圆环绕，缓慢横移；焦点在人物之间切换」

### 11.7 大师风格对变换矩阵的微调

选了大导演之后，§4.5.2 的基础运镜变换规则不是全部照搬——大师的签名风格会**覆盖/增强**特定运镜的参数：

```
选定大师：斯皮尔伯格（环绕移焦）
  影响的基础运镜：
    环绕 (Orbit) → 增强规则：
      轨迹：        全圆 → 限定为半圆 180 度
      焦点：        固定焦点 → 焦点在主体之间切换（移焦）
      速度：        匀速 → 慢入慢出，起幅落幅各留 2 秒
      追加描述：    "斯皮尔伯格式环绕移焦，镜头围绕人物做半圆平滑运动，
                    焦点在[人物A]和[人物B]之间自然切换"

选定大师：迈克尔·贝（低角旋转）
  影响的基础运镜：
    环绕 (Orbit) → 增强规则：
      机位高度：    平视 → 低机位，仰拍角度
      速度：        平滑 → 加速旋转，慢动作结束
      追加描述：    "迈克尔·贝式低机位英雄环绕，广角镜头，
                    以主体为中心快速半环绕，慢动作收尾"

选定大师：诺兰（推镜压迫）
  影响的基础运镜：
    推镜头 (Push In) → 增强规则：
      画幅：        常规 → IMAX 大画幅，全画幅无裁切
      速度：        平滑 → 极缓慢推进，制造时间拉伸感
      追加描述：    "诺兰式推镜，IMAX 全画幅，以极慢速度推进，
                    每一帧都在强化压迫感"
    拉镜头 (Pull Out) → 增强规则：
      目的：        展示环境 → 揭示时间/空间关系
      追加描述：    "回拉揭示全局，空间维度拉伸"

选定大师：王家卫（手持迷离）
  影响的基础运镜：
    手持晃动 (Handheld) → 增强规则：
      快门：        正常 → 慢速快门，拖影感
      色彩：        正常 → 霓虹光溢出，色彩饱和度增强
      追加描述：    "王家卫式手持，慢速快门轻微拖影，
                    霓虹灯光在画面边缘形成绿色/红色溢出"

选定大师：库布里克（单点透视）
  影响的基础运镜：
    推镜头 (Push In) → 增强规则：
      机位：        任意 → 必须正面中轴，对称构图
      轨迹：        可偏移 → 笔直推进，零偏移
      追加描述：    "库布里克式单点透视，镜头沿中轴线笔直推进，
                    画面严格对称"

选定大师：韦斯·安德森（横移平拍）
  影响的基础运镜：
    移镜头 (Track) → 增强规则：
      机位高度：    任意 → 必须 90 度正对主体
      角度：        有角度 → 平行横移，零角度偏移
      追加描述：    "韦斯·安德森式横移，镜头 90 度正对主体平行移动，
                    居中对称构图"

选定大师：姜文（急推急拉）
  影响的基础运镜：
    推镜头 (Push In) → 增强规则：
      速度：        平滑 → 极速推进/骤停
    拉镜头 (Pull Out) → 增强规则：
      速度：        平滑 → 瞬间拉出
      追加描述：    "姜文式急推/急拉，镜头运动带爆发力，
                    变焦结束瞬间定格"
```

**未选大师时**：使用 §4.5.2 的标准变换矩阵，无任何增强/覆盖。

**选了大导演时**：变换矩阵加载对应大师的增强规则，替换/追加描述短语。同时 DIM 追问中的映射检查会加载大师专属的前置条件。例如选了斯皮尔伯格后环绕的映射检查会增加：「移焦切换的落点清楚吗？两个焦点之间的距离不能太远。」

**大师风格与运镜的冲突规则**：

| 用户选的大师 | 用户选的运镜 | 行为 |
|-------------|-------------|------|
| 斯皮尔伯格 | 环绕 | 增强环绕，追加移焦 |
| 斯皮尔伯格 | 移镜头 | 保持标准变换 |
| 诺兰 | 推镜头 | 增强为 IMAX 慢推 |
| 诺兰 | 拉镜头 | 增强为空间揭示 |
| 王家卫 | 手持 | 增强，追加慢速快门 |
| 王家卫 | 推镜头 | 保持标准变换（王家卫不签名推镜） |
| 库布里克 | 推镜头 | 增强为单点透视推 |
| 库布里克 | 环绕 | 建议替换为单点推（库布里克风格不擅长环绕） |
| 韦斯·安德森 | 移镜头 | 增强为横移平拍 |
| 韦斯·安德森 | 推镜头 | 保持标准变换 |
| 姜文 | 推/拉 | 增强为急推急拉 |
| 姜文 | 移镜头 | 保持标准变换 |

## 12. 从现有项目借鉴的设计点

### 12.1 ai-shotlive → 上下文感知输出

**来源**：`github.com/VViles/ai-shotlive` — Phase 03 导演工作台

**借鉴点**：ai-shotlive 在生成首尾帧时，自动带入当前场景图 + 角色服装图作为条件，减少「不连戏」。

**落地到 NextShotPhoto**：

在输出每镜尾帧 prompt 时，附上一段「使用建议」提醒用户：

```
📌 首尾帧使用建议
本镜视频的首帧 = 上一镜尾帧图（已有）
本镜视频的尾帧 = 用本 prompt 生成的图
→ 推荐用 Seedance 首尾帧模式生成视频
→ 首帧图 + 尾帧图喂给同一任务，视频引擎会自动插值
```

如果 agent 能够感知到用户使用的视频引擎（如已配 Seedance API），可以进一步改为：

```
📌 首尾帧使用建议
检测到已配置 Seedance 2.0，推荐用首尾帧模式：
首帧：[上一镜尾帧图路径/URL]
尾帧：[本镜尾帧图路径/URL]
→ 两条图 + 本 prompt → 一段平滑过渡的视频
```

**目的**：不是生成图片，而是让用户明确知道「上一镜尾帧图 = 本镜首帧」这个关键连接，减少用户侧的操作断层。

---

### 12.2 videoclaw → 纯 prompt 层与执行层分离

**来源**：`github.com/hsaigroup/videoclaw` — Skills 模块分层设计

**借鉴点**：videoclaw 将 `video-text-storyboard`（分镜提示词生成）与 `video-i2v`（图生视频执行）拆成独立的 skill，职责清晰，互不耦合。

**落地到 NextShotPhoto**：

NextShotPhoto 的职责边界必须写死：

```
┌─────────────────────────────────────┐
│      NextShotPhoto 的职责范围        │
│                                     │
│  ✅ 接收/维护 NextShotContext        │
│  ✅ DIM 交互（解析+追问+映射检查）    │
│  ✅ 生成尾帧 prompt                  │
│  ✅ 字面量焊接 + diff 校验           │
│  ✅ 输出使用建议（提示用户连接执行层） │
│  ✅ [可选] 调用用户指定的生成工具      │
│                                     │
│  ❌ 不强制生成图片                    │
│  ❌ 不调用视频 API                    │
│  ❌ 不管理文件/路径/URL（仅记录路径）  │
└─────────────────────────────────────┘
```

每镜输出末尾固定加上：

```
🔄 下一步操作建议
① 用本 prompt 在你的图片生成工具中生成尾帧图
② 用上一镜尾帧图（首帧）+ 新生成的尾帧图 → Seedance 首尾帧模式 → 视频
③ 把新生成的尾帧图地址更新到镜头上下文中，以便下一镜使用
```

**目的**：不管下游用什么引擎（Seedance / Kling / Wan / 可灵），NextShotPhoto 只输出 prompt，不绑死任何执行路径。

---

### 12.3 StoryMem → 记忆结构验证

**来源**：`github.com/Kevin-thu/StoryMem` — 字节跳动开源，记忆条件化多镜视频生成

**借鉴点**：StoryMem 的核心创新是记忆包（Memory Pack）——每镜生成后自动提取关键帧特征，压缩成紧凑表示，传递给下一镜。这和我们 NextShotContext 的「锁定域跨镜传递」设计完全同构。

**落地到 NextShotPhoto**：

将现有的 `NextShotContext` 结构与 StoryMem 的 Memory Pack 做对应映射：

| StoryMem Memory Pack | NextShotContext | 说明 |
|----------------------|----------------|------|
| 角色外观 token | `lock.character` + `lock.clothing` | 人物身份锁定 |
| 场景环境 token | `lock.atmosphere` + `lock.lighting` | 氛围+光线锁定 |
| 记忆压缩机制 | 字面量焊接 + diff 校验 | 文本级压缩（非向量级） |
| 逐镜自动更新 | `shot.*` 字段每镜更新 | Delta 变化量 |
| 长程记忆保留 | `history[]` 数组 | 全历史可追溯 |

**关键推论**：既然 StoryMem 用向量级的 Memory Pack 能跑通跨镜一致，那我们用文本级的字面量焊接 + 变体隔离 + 首帧视觉锚定三重锁，在 prompt 层实现的效果应该足够覆盖大多数使用场景。StoryMem 解决的是模型层的问题（训练一个 M2V LoRA），我们解决的是 prompt 层的问题（写一个不出漂移的尾帧描述）。两个层面可以叠加使用——用户先用 NextShotPhoto 生成尾帧 prompt，再用 StoryMem 的 M2V 模式生视频。

**目的**：验证设计方向正确——记忆传递 + 跨镜锁定 + 逐镜 Delta 更新，在学术和工程实践中都有先例。

### 12.4 三个借鉴点的交叉关系

```
ai-shotlive 上下文感知
    ↓ 提示：输出的 prompt 应该附带使用场景建议
    ↓
videoclaw 职责分离
    ↓ 约束：NextShotPhoto 只做 prompt，不做执行
    ↓
StoryMem 记忆结构
    ↓ 验证：锁定域 + Delta 的设计方向有学术/工程先例
    ↓
三重锁定（4.1-4.3）
    ↓ 实现：字面量焊接 + 变体隔离 + 首帧锚定
    ↓
最终产出：一个不漂移、不越界、有上下文的尾帧 prompt
```

这三个借鉴点不是并列的可选功能，而是构成了一个**递进的约束链**：知道该输出什么（上下文）→ 知道不该做什么（职责分离）→ 知道为什么这么设计是对的（记忆结构验证）。
